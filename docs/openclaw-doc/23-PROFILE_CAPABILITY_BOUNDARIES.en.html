ACL Scenario

ACL is currently an experimental multi-agent memory isolation feature, and this video shows the current result directly.

ACL here means `Access Control List`, but the plain-language meaning on this page is: **experimental multi-agent memory isolation**. Once enabled, the current tested plugin path can narrow each agent to its own long-term memory by default. It should not yet be read as a fully hardened backend security boundary. So the film shows the full chain: which agents exist, how `alpha` stores a memory, and why `beta` still cannot read it.

The video now makes four steps explicit
  • It starts in `Agents` and proves `main / alpha / beta` exist as distinct scopes.
  • It enters `alpha` chat and shows a memory being stored there first.
  • It then switches to `beta`, where only beta-scoped memory remains visible.
  • It finally asks for alpha’s workflow and gets `UNKNOWN`.

If you want to know how to enable it, go straight to document 24. The real config lives under plugins.entries.memory-palace.config.acl.

Profiles In WebUI

What Profile B / C / D mean from the OpenClaw WebUI user’s point of view

This section deliberately avoids pretending that B / C / D each have their own dedicated pages. The more accurate statement is that users mostly keep seeing the same `Skills / Chat / tool card / recall block` surfaces, while the real differences emerge behind those surfaces and in output quality.

Two misleading framings this page refuses
  • It does not make Dashboard setup forms the main story for B / C / D.
  • It does not invent separate WebUI pages for differences that are mainly backend-path differences.
Profile A

The user still sees the same WebUI shell, but the retrieval boundary shrinks to the minimum validation tier.

A is not a separate simplified page. The same chat, skills, and agent shell remain visible, while the backend retrieval path falls back to keyword-only with no embedding and no reranker.

User-facing manifestationWhat not to over-claim
The plugin entry, chat write, and basic recall evidence are still visible.Do not describe it as the long-term recommended tier; it is closer to the minimum validation baseline.
Profile B

The user mostly sees “the chain works,” not an advanced-provider console

In WebUI terms, B feels like the baseline where chat, default recall, tool cards, and answer blocks are already alive. Durable memory is visibly entering the thread.

User-facing manifestationWhat not to over-claim
Visible recall blocks and a functional visual-memory loop are already present.Do not describe it as the final real-model quality tier or the long-term production target.
Profile C

The WebUI shape barely changes. The deeper change is recall and extraction quality.

For the user, C is not a new page. It is the same WebUI thread now running on a real embedding / reranker / model chain. The visible form is similar; the depth and reliability of recall improve.

The current public default also needs to stay explicit: `Profile C` guarantees `embedding + reranker` by default, while `write_guard / compact_gist / intent_llm` remain opt-in instead of default-on.

The public story now keeps only the screenshots that are still directly embedded by the public pages. Older `profile-matrix/*` inventory is maintainer-side evidence stock, not part of the default user-facing proof set of this page.

User-facing manifestationWhat not to over-claim
The same tool-card and answer-block surface becomes stronger because the underlying provider chain is real.Do not treat “env configured” as equivalent to “provider actually healthy”; the probe still has to be green.
Profile D

The user still sees something very close to C, but the default target becomes the full advanced surface.

D usually does not grow a distinct WebUI surface. The more accurate interpretation is that the user still sees the same chat-and-tool shell, while the default target becomes `embedding + reranker + write_guard / compact_gist / intent_llm` together.

Provider location can still be local, intranet, or remote. The real boundary is provider health, network conditions, and latency budget rather than “remote by definition.”

The current public default is now: `Profile D` includes `write_guard / compact_gist / intent_llm` by default, while `plugins.slots.memory` still stays on `memory-palace` and `memory-core` remains only a facade-compat shim.

User-facing manifestationWhat not to over-claim
It still looks close to C in the WebUI, but the default target is the full advanced surface.Do not describe it as “remote-only,” and do not describe it as naturally low-latency.
One-line summary

A / B / C / D differ more in backend depth than in top-level WebUI shape

That is why this page starts from the user-visible evidence first and only then explains where the differences remain behind the scenes. For users, the first thing that matters is not the profile label. It is whether `skill entry / recall block / tool card / answer block` have already entered the thread they use every day.
