# Memory Palace Troubleshooting

This document helps you quickly identify and resolve common issues when using Memory Palace.

---

## 1. Frontend 502 or Interface Timeout

**Symptom**: The page loads but lists are empty or interfaces return errors.

> There is a very common "looks broken but is actually normal gating" scenario in the current version:
>
> - The page redirects to `/setup`
> - The top-right corner shows `Set API key`
> - `Memory / Review / Maintenance / Observability` displays empty states, waiting prompts, or `401`
>
> This usually does not mean the frontend is down. Instead, **you have not yet authorized the protected interfaces**.
>
> If you are using **Docker one-click deployment**, you do not need to manually click this button by default. First confirm whether you used the Docker env file generated by `apply_profile.*` / `docker_one_click.*`.

**Troubleshooting Steps**:

1. Confirm the **backend is running**:

   ```bash
   curl -fsS http://127.0.0.1:8000/health
   ```

   > The backend health check endpoint `GET /health` returns `status`, `index`, `runtime`, and other fields (see the `health()` function in `backend/main.py`).

2. Confirm the **frontend proxy target is correct**:

   Check the `apiProxyTarget` value in `frontend/vite.config.js`:

   ```javascript
   // Default target: http://127.0.0.1:8000
   const apiProxyTarget =
     process.env.MEMORY_PALACE_API_PROXY_TARGET ||
     process.env.NOCTURNE_API_PROXY_TARGET ||
     'http://127.0.0.1:8000'
   ```

   If the backend is running on a different port, start the frontend from the
   `frontend/` directory with the environment variable set:

   ```bash
   cd frontend
   MEMORY_PALACE_API_PROXY_TARGET=http://127.0.0.1:9000 npm run dev
   ```

3. In **Docker scenarios**, confirm port mapping:

   - Default backend port: `18000` (mapped to container's `8000`)
   - Default frontend port: `3000` (mapped to container's `8080`)
   - Configurable via `MEMORY_PALACE_BACKEND_PORT` and `MEMORY_PALACE_FRONTEND_PORT` environment variables (see `docker-compose.yml`)

4. Check backend logs:

   ```bash
   # When started locally (uvicorn/python run_sse.py), check the current terminal output first
   # For Docker deployments, view container logs
   docker compose -f docker-compose.yml logs backend --tail=50
   ```

## 2. `/maintenance/*`, `/review/*`, or `/browse/*` Returns 401

**Cause**: `MCP_API_KEY` is enabled but the request does not include an authentication header. Note that read operations on `/browse/node` are also protected.

**Troubleshooting and Resolution**:

- **Method 1**: Add authentication header to curl:

  ```bash
  curl -fsS http://127.0.0.1:8000/maintenance/orphans \
    -H "X-MCP-API-Key: <YOUR_MCP_API_KEY>"
  ```

- **Method 2**: Use Bearer format:

  ```bash
  curl -fsS http://127.0.0.1:8000/maintenance/orphans \
    -H "Authorization: Bearer <YOUR_MCP_API_KEY>"
  ```

- **Frontend**: Inject `window.__MEMORY_PALACE_RUNTIME__` (see [SECURITY_AND_PRIVACY.en.md](SECURITY_AND_PRIVACY.en.md) Section 4)
- **Frontend page**:
  - If you see `/setup`, follow the page prompts to select `basic + Profile B`, then click `Apply`
  - If the page then shows `Restart Backend`, click it again to complete the locally controlled restart
  - If the backend already has `MCP_API_KEY` configured, you can also click `Set API key` / `Update API key` in the top-right corner

An important clarification to avoid conflating two separate keys:

- The `Set API key / Update API key` in the top-right corner
  - Primarily resolves access to Dashboard protected interfaces
  - On the current loopback / same-origin page path, the frontend also attaches it to `/bootstrap/provider-probe`, `/bootstrap/apply`, and `/bootstrap/restart`
- `status / apply / restart` within `/setup`
  - Still uses `/bootstrap/*`'s own loopback / same-origin gating
  - If the backend has `MCP_API_KEY` configured
    - This path must also carry that key
  - If the backend does not have `MCP_API_KEY` configured
    - Loopback write operations can still proceed
    - But the backend will log a warning indicating bootstrap write requests are not key-protected

- **Local debugging**: You can set the insecure local override (loopback only):

  ```bash
  # Add to .env
  MCP_API_KEY_ALLOW_INSECURE_LOCAL=true
  ```

**Determine the specific cause based on the returned `reason` field (the current auth helper is defined in `backend/api/maintenance_common.py`; `maintenance.py` / `browse.py` / `review.py` all reuse it):**

| `reason` | Meaning | Action |
|---|---|---|
| `invalid_or_missing_api_key` | Key is incorrect or not provided | Check if the key is correct |
| `api_key_not_configured` | `MCP_API_KEY` is empty in `.env` during local manual startup | Set the key or enable insecure local |
| `insecure_local_override_requires_loopback` | Insecure local is enabled but request is non-loopback | Ensure access from `127.0.0.1` or `localhost` |

> If you see any of the following:
>
> - `Awaiting Input`
> - `Failed to load node`
> - `Connection Lost`
> - `maintenance_auth_failed | api_key_not_configured`
>
> Configure the key first, then investigate whether there are other issues.
>
> Additionally: the Dashboard API key manually entered in the frontend is now stored only in the current page's memory (memory-only). It is cleared when the tab is closed or refreshed. Use `window.__MEMORY_PALACE_RUNTIME__` injection for persistent auth.

## 3. SSE Startup Failure or Port Conflict

**Symptom**:

- Locally starting `python run_sse.py` reports `address already in use`
- Or accessing `http://127.0.0.1:3000/sse` fails under Docker

**Resolution**:

1. Change the port (SSE defaults to port `8000`, see `backend/run_sse.py` line 105):

   ```bash
   HOST=127.0.0.1 PORT=8010 python run_sse.py
   ```

2. Or find and release the occupied port:

   ```bash
   # macOS / Linux
   lsof -i :8000
   kill -9 <PID>
   ```

   ```powershell
   # Windows PowerShell
   netstat -ano | findstr :8000
   taskkill /PID <PID> /F
   ```

3. For Docker one-click deployment, check the frontend entry first:

   ```bash
   curl -i -H 'Accept: text/event-stream' http://127.0.0.1:3000/sse
   ```

   Under normal conditions you should see:

   ```text
   event: endpoint
   data: /messages/?session_id=...
   ```

4. If you are **accessing `run_sse.py` from another machine**:

   - Do not continue using `HOST=127.0.0.1`
   - Change to `HOST=0.0.0.0` (or your actual bind address)
   - Also set up `MCP_API_KEY`, firewall, reverse proxy, and other network-side protections

A clarification on current terminology to avoid conflating two paths:

- `python run_sse.py`
  - Is the standalone SSE debugging path
- `uvicorn main:app`
  - Now also directly mounts `/sse`, `/messages`, and `/sse/messages`
- Docker / frontend proxy
  - Defaults to using the backend-embedded SSE; there is no separate `sse` container

### Remote Access to `/sse` Returns `421 Misdirected Request`

**Symptom**:

- Works locally but returns `421` when accessed from another machine
- Or you manually sent an incorrect `Host` header and the request was rejected

**Resolution**:

1. First check if `HOST` is still set to loopback:

   ```bash
   HOST=127.0.0.1 PORT=8010 python run_sse.py
   ```

   This form is only intended for local access. For remote access, change to:

   ```bash
   HOST=0.0.0.0 PORT=8010 python run_sse.py
   ```

2. If you are indeed doing local loopback debugging but received `421` / `403`:

   - First check if the `Host` / `Origin` in the request matches the current address
   - Then check if the request passed through a proxy that rewrites headers

3. If the backend output does **not** contain a Python traceback and you only see `421` / `403`:

   - This usually does not mean the service crashed
   - Rather, the `Host` / `Origin` validation rejected the request as expected

## 5. Sudden Drop in Search Quality

**Troubleshooting Steps**:

1. **Check `degrade_reasons`**: The `degrade_reasons` field returned by the `search_memory` MCP tool tells you the specific reason for search pipeline degradation. Common values include:

   | `degrade_reasons` Value | Meaning | Source File |
   |---|---|---|
   | `embedding_fallback_hash` | Embedding API unreachable, fell back to local hash | `backend/db/sqlite_client.py` |
   | `embedding_config_missing` | Embedding configuration missing | `backend/db/sqlite_client.py` |
   | `embedding_request_failed` | Embedding API request failed | `backend/db/sqlite_client.py` |
   | `reranker_request_failed` | Reranker API request failed | `backend/db/sqlite_client.py` |
   | `reranker_config_missing` | Reranker configuration missing | `backend/db/sqlite_client.py` |
   | `compact_gist_llm_empty` | Compact Gist LLM returned empty result | `backend/mcp_server.py` |
   | `index_enqueue_dropped` | Index task enqueue was dropped | `backend/mcp_server.py` |

   > `write_guard_exception` belongs to the write/learn pipeline (e.g., `create_memory`, `update_memory`, explicit learn triggers). Its semantics indicate that the write was fail-closed rejected; it is not a search quality degradation.
   >
   > If `embedding_fallback_hash` appears here, the current payload will also carry `semantic_search_unavailable=true`. In this case you may still get results, but the correct understanding is "keyword / fallback search is still alive," not "semantic recall is still working normally."
   >
   > An additional note to avoid misinterpreting old logs or skips as plugin failures:
   >
   > - In older versions, if the host omitted `ctx` on the `message:preprocessed` hook, you might see `Hook error [message:preprocessed]`
   > - The current version normalizes the missing `ctx` as an empty object on this path; this phenomenon itself should no longer be treated as a plugin failure
   > - When auto-capture encounters `Skipped: write_guard blocked create_memory` / `update_memory` collisions, it typically means the candidate content was normally skipped, not that the plugin has entirely failed

2. **Check Embedding / Reranker API reachability**:

   ```bash
   # Configuration semantics: RETRIEVAL_EMBEDDING_BACKEND only controls embedding.
   # There is no RETRIEVAL_RERANKER_BACKEND for reranker; to force using your own reranker API locally,
   # explicitly set RETRIEVAL_RERANKER_ENABLED=true with RETRIEVAL_RERANKER_API_BASE/API_KEY/MODEL.
   # Note: RETRIEVAL_*_API_BASE may already include /v1; avoid manually appending /v1 again
   # Testing actual call endpoints is more accurate for health checks:
   curl -fsS -X POST <RETRIEVAL_EMBEDDING_API_BASE>/embeddings \
     -H "Content-Type: application/json" \
     -d '{"model":"<RETRIEVAL_EMBEDDING_MODEL>","input":"ping"}'
   curl -fsS -X POST <RETRIEVAL_RERANKER_API_BASE>/rerank \
     -H "Content-Type: application/json" \
     -d '{"model":"<RETRIEVAL_RERANKER_MODEL>","query":"ping","documents":["pong"]}'
   ```

   > **Recommended troubleshooting order**:
   > - First confirm whether you are using the `router` approach or directly configuring `RETRIEVAL_EMBEDDING_*`, `RETRIEVAL_RERANKER_*`, `WRITE_GUARD_LLM_* / COMPACT_GIST_LLM_*` individually.
   > - If using direct configuration, check the actual `*_API_BASE` / `*_API_KEY` / `*_MODEL` first.
   > - If using the `router` approach, then check the `ROUTER_*` configuration and the router service itself.

3. **Rebuild the index** (via MCP tool call):

   ```python
   # Rebuild index
   rebuild_index(wait=true)
   # Check index status
   index_status()
   ```

4. **View the observability summary** (via HTTP API):

   ```bash
   curl -fsS http://127.0.0.1:8000/maintenance/observability/summary \
     -H "X-MCP-API-Key: <YOUR_MCP_API_KEY>"
   ```

5. **Check gist-audit background jobs** (if you triggered a gist quality rerun from Maintenance):

   ```bash
   curl -fsS -X POST http://127.0.0.1:8000/maintenance/gist-audit/run \
     -H "X-MCP-API-Key: <YOUR_MCP_API_KEY>"

   curl -fsS http://127.0.0.1:8000/maintenance/gist-audit/job/<JOB_ID> \
     -H "X-MCP-API-Key: <YOUR_MCP_API_KEY>"
   ```

   - `run` now returns `status=queued` and a `job_id` immediately
   - the `gist_audit` field inside `observability/summary` is aggregate-only; use `job/{job_id}` for one job's live status

6. **Check configuration parameters**: Confirm `RETRIEVAL_RERANKER_WEIGHT` is within a reasonable range (`.env.example` comments recommend `0.20 ~ 0.40`, default `0.25`)

7. **Do not panic when you see new fields in the observability page**:

   - `scope_hint`: Simply tells the search "which scope to prioritize"
   - `sm-lite`: A new set of lightweight runtime states in the current version, not an error
   - `Runtime Snapshot`: A troubleshooting summary; not every field needs to have a value
   - `gist_audit`: Aggregate gist-quality feedback stats; it does not mean the current request is waiting on a live audit

## 8. Database Migration Errors

**Symptom**: Migration lock timeout at startup, similar to `Timed out waiting for migration lock`.

**Background**: Memory Palace uses a file-lock-based migration mechanism (see `backend/db/migration_runner.py`) to prevent multiple processes from executing migrations simultaneously.

**Troubleshooting and Resolution**:

1. **Check if multiple processes are starting simultaneously**

2. **Adjust lock timeout**: Set in `.env` (default `10` seconds):

   ```bash
   DB_MIGRATION_LOCK_TIMEOUT_SEC=30
   ```

3. **Manually specify lock file path**:

   ```bash
   DB_MIGRATION_LOCK_FILE=/tmp/memory_palace.migrate.lock
   ```

   > If not set, the default lock file is `<database-file>.migrate.lock` (e.g., `demo.db.migrate.lock`), saved in the same directory as the database file.

4. **Manually delete residual lock file and restart**:

   ```bash
   # Find and delete the lock file (default location is next to the database file)
   rm -f /path/to/demo.db.migrate.lock
   ```

**Verification anchor**: The `backend/tests/test_migration_runner.py` in the repository covers migration lock and timeout scenarios.

## 10. CORS Error (Cross-Origin Access Denied)

**Symptom**: Browser reports a CORS error when the frontend requests the backend API.

**Explanation**: The current default **does not allow all Origins**. When `CORS_ALLOW_ORIGINS` is left empty, the backend only allows common local origins:

```python
http://localhost:5173
http://127.0.0.1:5173
http://localhost:3000
http://127.0.0.1:3000
```

**If you still get errors**, common causes include:

- The frontend dev server proxy is not configured correctly (check `frontend/vite.config.js`)
- In Docker deployment, the frontend Nginx is not correctly forwarding to the backend (check `deploy/docker/nginx.conf.template`)
- You are accessing the backend from a **browser origin not on the allowlist**

**Recommended Actions**:

- Local development:
  - Keep `CORS_ALLOW_ORIGINS=` empty
- Production browser access:
  - Set `CORS_ALLOW_ORIGINS` explicitly to your frontend address list
  - Example: `CORS_ALLOW_ORIGINS=https://app.example.com,https://admin.example.com`
- Do not use `*` for convenience
  - Especially when you need credentials / cookie / auth headers

## 11. External Import Returns `rate_limit_state_unavailable` / `state_lock_timeout`

**Symptom**: External import request is rejected, and the response shows:

- `reason=rate_limit_state_unavailable`
- `rate_limit_state_error=state_lock_timeout`

**In plain terms**: The shared rate-limit state file failed to acquire its lock in time.

This usually does not mean "the file is corrupted." More commonly:

- You have shared rate limiting enabled
- Multiple processes are accessing this state file at the same time
- The default wait time is insufficient for the current machine/disk/concurrency level

**Troubleshooting and Resolution**:

1. First confirm you are indeed using a shared state file:

   ```bash
   EXTERNAL_IMPORT_RATE_LIMIT_STATE_FILE=/path/to/external_import_rate_limit.json
   EXTERNAL_IMPORT_REQUIRE_SHARED_RATE_LIMIT=true
   ```

2. If this pipeline is inherently multi-process concurrent, increase the lock wait time:

   ```bash
   EXTERNAL_IMPORT_RATE_LIMIT_STATE_LOCK_TIMEOUT_SECONDS=3.0
   ```

   Common starting values to try:

   - `3.0`
   - `5.0`

3. Also confirm that `EXTERNAL_IMPORT_RATE_LIMIT_STATE_FILE` points to:
   - A regular file in a locally writable directory
   - Not a directory
   - Not a read-only mount

4. If you do not actually need cross-process shared rate limiting:
   - You can remove `EXTERNAL_IMPORT_REQUIRE_SHARED_RATE_LIMIT=true`
   - Or simply do not configure `EXTERNAL_IMPORT_RATE_LIMIT_STATE_FILE` at all

An additional note on rejection reasons from this round of security hardening, to avoid misinterpreting them as "the file is corrupted" on first encounter:

- `symlink_not_allowed`
  - More precisely: "the import target itself is a symlink, and the current policy directly rejects it"
- `file_changed_during_validation`
  - More precisely: "the file was replaced between validation and actual opening"
  - Currently treated as a risk; reading does not continue
- `path_not_allowed`
  - More precisely: "the import target initially appeared to be within the allowed root directory, but the final resolved path is outside it"
  - Currently fail-closes immediately; reading does not continue

**Additional note**:

- `EXTERNAL_IMPORT_RATE_LIMIT_STATE_LOCK_TIMEOUT_SECONDS` only affects how long this state file lock waits
- It does not change the main rate-limit window, nor does it relax `allowed_roots / allowed_exts / max_files / max_total_bytes`

## 12. Backend Startup Reports `No module named 'diff_match_patch'`

**Symptom**: You see `ModuleNotFoundError: No module named 'diff_match_patch'` when starting `backend/main.py`, `run_sse.py`, or before/after calling `/review/diff`.

**Current Actual Behavior**:

- `diff_match_patch` is now an optional dependency
- If present, `/review/diff` preferentially returns semantic HTML diff
- If absent, the backend automatically falls back to `difflib.HtmlDiff` table diff and `difflib.SequenceMatcher` summaries
- So the absence of this package should no longer kill the backend on startup

**Troubleshooting and Resolution**:

1. First confirm you are actually running this repository's code:

   ```bash
   cd backend && python - <<'PY'
   from api.utils import get_text_diff
   html, unified, summary = get_text_diff("old\n", "new\n")
   print("<table class=\"diff\"" in html, "--- old_version" in unified, summary)
   PY
   ```

2. If you want to restore the better semantic diff display, install the optional package:

   ```bash
   cd backend && pip install diff-match-patch
   ```

3. If you are still exiting during startup due to this error:
   - First confirm you are not running an old venv, old Docker image, or old wheel
   - Then confirm the local process is not pointing to a different repository path

**Boundary note**: The fallback only guarantees service availability and an unbroken review pipeline; semantic diff display quality is still better with `diff_match_patch`.

## 13. Getting Help

If the steps above cannot resolve your issue:

1. Check the full backend logs: locally, check the startup terminal output; for Docker, use `docker compose -f docker-compose.yml logs backend --tail=200`
2. Check the `status` and `index` fields returned by `GET /health`
3. View the system overview via `GET /maintenance/observability/summary` (this endpoint is protected by `MCP_API_KEY`; include `X-MCP-API-Key` or `Authorization: Bearer`)
4. When submitting an Issue, please include: error message, operating system, Python version, Node.js version, and the Profile being used
